PHẦN 1: CHUẨN BỊ "ĐỒ NGHỀ" (CÀI ĐẶT MÔI TRƯỜNG)

Chào mừng! Trước khi xây nhà, chúng ta cần búa, cưa, đinh. Đây là 3 "món đồ nghề" cơ bản bạn cần chuẩn bị **trước khi** bắt đầu học. Ai không hoàn thành PHT 00 sẽ không thể thực hành.

1.2. Nhiệm vụ 1: Tài khoản GitHub - "Facebook của Dân Code"

GitHub là gì? Nó là một trang web, một mạng xã hội, nhưng thay vì "post" ảnh, bạn sẽ "post" code của mình lên đó. Nó giống như "Google Drive" vậy, là nơi bạn **sao lưu (backup)** code của mình an toàn.

Tại sao phải chuyên nghiệp? Khi đi phỏng vấn, nhà tuyển dụng sẽ hỏi "Username GitHub của em là gì?". Họ sẽ vào xem code của bạn. Vì vậy, hãy chọn một cái tên nghiêm túc (ví dụ: `nguyenvana`, `an-nguyen-dev`) và tránh những tên như `beheo123`.

Hành động: Truy cập https://github.com/, chọn "Sign up", và làm theo hướng dẫn. Nhớ vào email để **xác thực** tài khoản!

1.3. Nhiệm vụ 2: VS Code - "Cây búa" xịn

VS Code là gì? Nó là một trình soạn thảo văn bản (Text Editor), giống như Notepad, nhưng "xịn" hơn 1000 lần. Nó giúp chúng ta viết code bằng cách tô màu cú pháp, gợi ý lệnh, v.v.

Hành động: Truy cập https://code.visualstudio.com/, tải về và cài đặt. (Lưu ý cho Windows: Hãy **tick vào 4 ô** ở màn hình "Select Additional Tasks" khi cài đặt).

1.4. Nhiệm vụ 3: Git - "Cỗ máy thời gian"

Đây là "trái tim" của môn học. GitHub là trang web (dịch vụ), còn Git là phần mềm (công cụ) chạy trên máy của bạn.

  • Đối với Windows: Truy cập https://git-scm.com/downloads. Khi cài đặt, đừng hoảng sợ! Chỉ cần bấm **"Next" ... "Next" ... "Next"** cho đến khi "Install".
  • Đối với macOS: Mở "Terminal", gõ `git --version`. macOS sẽ tự động hỏi bạn "Bạn có muốn cài đặt 'command line developer tools' không?". Click **"Install"** và ngồi chờ.

1.5. Nhiệm vụ 4: Kiểm tra (Nộp PHT 00)

Làm sao biết bạn đã thành công? Hãy mở Terminal (hoặc **Git Bash Here** trên Windows) và gõ lệnh sau:

git --version

Nếu nó hiển thị một phiên bản (ví dụ: `git version 2.40.1...`), bạn đã SẴN SÀNG.

PHẦN 2: CỖ MÁY THỜI GIAN (LÀM VIỆC CỤC BỘ)

Mục tiêu của phần này là hiểu Git là gì và thực hiện 3 lệnh quan trọng nhất: `init`, `add`, và `commit`.

2.1. Vấn đề "Muôn thuở": Save As...

Bạn có quen với việc này không? `BaoCao_v1.docx`, `BaoCao_v2_final.docx`, `BaoCao_final_NopThay.docx`... Cách làm này RẤT LỘN XỘN. Chúng ta không biết ai sửa, sửa cái gì, và file nào là "chuẩn".

Giải pháp: Git là một "cỗ máy thời gian". Nó không lưu 10 file, nó chỉ lưu **những thay đổi**. Mỗi lần "lưu", Git "chụp một bức ảnh" (snapshot) dự án, cho phép bạn quay lại bất kỳ thời điểm nào.

2.3. Lý thuyết quan trọng nhất: 3 Vùng (The 3-Stage Workflow)

Git có 3 "vùng" làm việc, giống như 3 bước để chụp ảnh:

1. Working Directory (Nơi bạn code)
→
2. Staging Area (Nơi "chờ" chụp ảnh)
→
3. Local Repository (Cuốn Album - `.git`)
  1. Working Directory: "Phòng làm việc" bừa bộn của bạn, nơi bạn đang gõ code.
  2. Staging Area: "Phòng chờ". Bạn dùng lệnh `git add [tên file]` để đưa file đã "sạch sẽ" vào đây.
  3. Local Repository: "Cuốn Album". Bạn dùng `git commit -m "Thông điệp"` để "Tách!", chụp lại khoảnh khắc từ Staging vào đây.

2.4. Thực hành (Live Demo - PHT 1)

Mở Terminal `Git Bash` và làm theo:

# Bước 1: Cấu hình "Danh tính" (Làm 1 lần)
git config --global user.name "Nguyen Van A"
git config --global user.email "a.nguyen@email.com"

# Bước 2: Khởi tạo Dự án
mkdir my-portfolio-simple
cd my-portfolio-simple
git init

# Bước 3 & 4: Làm việc và Kiểm tra (Tạo file index.html)
git status

# Bước 5: Đưa vào "Phòng chờ"
git add index.html
git status

# Bước 6: Chụp ảnh (Commit)
git commit -m "Commit dau tien: Tao file index.html"

# Bước 7: Xem lại "Cuốn Album"
git log

PHẦN 3: GITHUB & BẢN SAO LƯU (LÀM VIỆC TỪ XA)

Mục tiêu: Đẩy "cuốn album" (Local Repo) của bạn lên "Google Drive cho code" (GitHub). Việc này dùng 2 lệnh: `remote` và `push`.

3.1. Vấn đề "Muôn thuở" (Phần 2)

"Cuốn album" của bạn đang nằm trên MÁY TÍNH CỦA BẠN. Nếu máy hỏng? MẤT HẾT! Chúng ta cần một "bản sao lưu" (backup) trên mạng. Đó chính là GitHub.
  • Local Repo: Kho chứa trên máy bạn.
  • Remote Repo: Kho chứa trên GitHub.

3.3. Thực hành (Live Demo - PHT 2)

Bước 1: Tạo Kho chứa (Repo) rỗng trên GitHub

  1. Đăng nhập GitHub, bấm dấu `+` -> "New repository".
  2. Tên repo: `my-portfolio-simple`.
  3. Chọn "Public".
  4. **QUAN TRỌNG:** **KHÔNG** tick vào 3 ô: README, .gitignore, license.
  5. Bấm "Create repository".

Bước 2: Kết nối Local với Remote

GitHub sẽ cho bạn 2 dòng lệnh. Copy dòng đầu tiên vào Terminal của bạn:

# "origin" là "biệt danh" cho URL của GitHub
git remote add origin https://github.com/[TEN-CUA-BAN]/my-portfolio-simple.git

Bước 3: Đẩy "Album" lên (Push)

Copy dòng lệnh thứ hai. Lệnh này "đẩy" code từ nhánh `main` ở máy bạn lên `origin` (GitHub).

git push -u origin main

Bước 4: Kiểm tra (Nộp PHT 2)

Tải lại (F5) trang GitHub của bạn. BOOM! File `index.html` và commit đã ở trên GitHub.

PHẦN 4: PHÂN NHÁNH (BRANCHING) - QUY TẮC LÀM VIỆC AN TOÀN

Đây là phần quan trọng nhất. Mục tiêu: Hiểu "tại sao" phải phân nhánh. Thực hiện `branch`, `checkout` và commit trên nhánh mới.

4.1. Vấn đề "Nguy hiểm": Sửa code trên "Bản gốc"

Quy tắc VÀNG:

TUYỆT ĐỐI KHÔNG BAO GIỜ CODE TRỰC TIẾP TRÊN NHÁNH `MAIN`!

Nhánh `main` là "bản chính thức", "bản sạch". Nếu bạn code dở dang trên `main`, sếp đòi demo, bạn sẽ gửi code lỗi!

4.2. Giải pháp: Phân nhánh (Branching) - "Photocopy"

Nhánh `main` là "Bản luận văn gốc". Bạn muốn thử viết "Lời kết" mới? Hãy **photocopy** (`branch`) bản gốc ra, đặt tên là `thu-viet-ket-bai`. Bạn thoải mái sửa trên bản photo này. Nếu hay thì gộp (Phần 5), dở thì vứt đi. "Bản luận văn gốc" (`main`) luôn an toàn.
main
...
→
them-so-thich

4.3. Thực hành (Live Demo - PHT 3)

Case Study: "Thêm mục 'Sở thích' vào Portfolio."

# Bước 1: Tạo "Bản photo" (Tạo nhánh)
git checkout main
git branch them-so-thich

# Bước 2: Di chuyển sang "Bản photo"
git checkout them-so-thich

# Mẹo: gộp 2 bước trên bằng `git checkout -b them-so-thich`

# Bước 3 & 4: Làm việc và Lưu "Bức ảnh" trên nhánh mới
(Sửa file index.html, thêm 
So thich
...)
git add .
git commit -m "Hoan thanh: Them muc So thich"

# Bước 5: "Điều kỳ diệu" (Thử quay lại main)
git checkout main
(Code "Sở thích" BIẾN MẤT!)
git checkout them-so-thich
(Code "Sở thích" XUẤT HIỆN TRỞ LẠI!)

# Bước 6: Xem "Sơ đồ" (Nộp PHT 3)
git log --graph --oneline --all

PHẦN 5: PULL REQUEST - QUY TRÌNH LÀM VIỆC NHÓM

Mục tiêu: Biết cách "gộp" code từ nhánh phụ (`them-so-thich`) về nhánh chính (`main`) một cách an toàn bằng Pull Request (PR).

Quy tắc VÀNG số 2:

KHÔNG BAO GIỜ TỰ Ý MERGE VÀO MAIN!

Việc gộp code bừa bãi có thể gây lỗi. Luôn luôn phải có người "review" (xem xét) code của bạn.

5.2. Giải pháp: Pull Request (PR) - "Yêu cầu Gộp code"

Pull Request (PR) là gì? Nó **KHÔNG** phải là lệnh Git. Nó là **tính năng của GitHub**.

PR là một "Yêu cầu" bạn gửi cho Sếp: "Sếp ơi, em đã làm xong 'bản nháp' `them-so-thich` và đẩy lên GitHub. Sếp làm ơn **Review** (xem) code của em, nếu ổn thì sếp hãy ấn nút **Merge** (gộp) nó vào `main` giùm em."
1. Push Nhánh Nháp (Đẩy `them-so-thich` lên GitHub)
→
2. Mở Pull Request (Trên web GitHub)
→
3. Review & Merge (Sếp duyệt và gộp vào `main`)

5.3. Thực hành (Live Demo - PHT 4)

Bước 1: Đẩy "Bản nháp" lên GitHub

git checkout them-so-thich
git push -u origin them-so-thich

Bước 2: Tạo Pull Request (PR) trên GitHub

  1. Vào repo trên GitHub, bạn sẽ thấy thanh màu vàng.
  2. Click **"Compare & pull request"**.
  3. Đảm bảo `base: main` <--- `compare: them-so-thich`.
  4. Viết Tiêu đề, click **"Create pull request"**.

Bước 3 & 4: Tự Review, Merge và Dọn dẹp

  1. (Đóng vai "Sếp") Mở tab "Files changed" để xem code.
  2. Quay lại "Conversation", click **"Merge pull request"** -> "Confirm merge".
  3. Sau khi gộp, click **"Delete branch"** để xóa nhánh `them-so-thich` trên GitHub.

Kết quả (Nộp PHT 4): Chụp lại màn hình PR có chữ **"Merged"** màu tím.

PHẦN 6: ĐỒNG BỘ & XUNG ĐỘT (WORKFLOW HÀNG NGÀY)

Mục tiêu: Biết cách "cập nhật" code mới nhất (`pull`) và biết cách "gỡ rối" (`conflict`).

Quy tắc VÀNG số 3:

LUÔN LUÔN `GIT PULL` TRƯỚC KHI BẮT ĐẦU CODE!

Code trên máy bạn có thể đã bị "cũ" so với server. Lệnh `git pull` giúp "tải" code mới nhất về.

6.2. Lệnh `git pull` - Cập nhật Code

Sau khi `merge` ở PHT 4, `main` trên GitHub đã mới. `main` trên máy bạn bị cũ. Hãy cập nhật:

# Chuyển về nhánh main
git checkout main
# "Kéo" (pull) code mới nhất từ origin về
git pull origin main

(Bạn sẽ thấy code "Sở thích" xuất hiện ở `main` trên máy).

6.3. Vấn đề "Trùm cuối": Xung đột (Merge Conflict)

Xung đột xảy ra khi nào? Khi 2 người cùng sửa **ĐÚNG MỘT DÒNG CODE** trong cùng một file. Khi bạn `pull` code mới về, Git "bối rối" không biết chọn code của AI. Nó sẽ "báo lỗi" và nhờ bạn "gỡ rối".

6.4. Thực hành (Live Demo - PHT 5 - "Diễn tập Gỡ rối")

Chúng ta sẽ "dàn cảnh" một vụ xung đột.

  1. (Đóng vai "Đồng nghiệp") Sửa code trên GitHub: Mở `index.html` trên GitHub, Sửa dòng `

    ` thành `

    ... (DA SUA TREN GITHUB)

    ` và Commit.
  2. (Đóng vai "Bạn") Sửa code ở Local: Mở `index.html` trên máy, **KHÔNG `pull`**, sửa **ĐÚNG DÒNG `

    `** thành `

    ... (DA SUA O LOCAL)

    ` và Commit.
  3. Gây ra Xung đột (The Explosion!): Gõ `git pull origin main`. Terminal sẽ báo **"BOOM! CONFLICT!"**.
  4. "Gỡ rối" (Resolve Conflict): Mở file `index.html`. Bạn sẽ thấy:
    <<<<<<< HEAD <h1>... (DA SUA O LOCAL)</h1> ======= <h1>... (DA SUA TREN GITHUB)</h1> >>>>>>> [dãy số hash]
  5. Nhiệm vụ của bạn: XÓA HẾT các dấu `<<<<<`, `=====`, `>>>>>` và sửa lại dòng `

    ` theo ý bạn muốn (ví dụ: `

    ... (DA GIAI QUYET XONG)

    `). Lưu file lại.
  6. Báo cho Git là "Đã gỡ rối xong": 
    git add .
git commit
(Lưu và đóng trình soạn thảo hiện ra)
git push origin main

PHẦN 7: TỔNG KẾT & BÀI TẬP LỚN (`clone` VÀ `GitHub Pages`)

Mục tiêu: Tổng hợp tất cả kỹ năng. Học thêm 2 kỹ năng mới `git clone` và `GitHub Pages`.

7.1. Kỹ năng mới: `git clone` - "Mượn" dự án

Thay vì `git init` (tạo dự án mới), `git clone [URL]` cho phép bạn "photocopy" toàn bộ dự án có sẵn của người khác về máy bạn.

7.2. Kỹ năng mới: GitHub Pages - Biến Code thành Web

Một tính năng miễn phí của GitHub cho phép biến kho code HTML/CSS của bạn thành một trang web **ĐANG CHẠY** trên Internet.

7.3. Thực hành (Live Demo - PHT Final)

  1. Clone dự án mẫu: 
    git clone https://github.com/jigar-sable/Portfolio-Website
cd Portfolio-Website
  2. Cá nhân hóa: Mở code và sửa `index.html` (thay tên, ảnh, text...).
  3. "Bẻ lái" Remote:
    • Tạo một repo MỚI, RỖNG trên GitHub của bạn tên `my-portfolio-final`.
    • Copy URL repo mới của bạn.
    • Gõ lệnh: 
      git remote set-url origin https://github.com/[TEN-CUA-BAN]/my-portfolio-final.git
  4. Lưu và Đẩy "Phiên bản của Tôi": 
    git add .
git commit -m "Final Project: Ca nhan hoa portfolio"
git push -u origin main
  5. Xuất bản (Deploy) với GitHub Pages:
    • Trên repo GitHub của bạn, vào `Settings` -> `Pages`.
    • Mục `Source`, chọn `Deploy from a branch`.
    • Mục `Branch`, chọn `main` (hoặc `master`), thư mục `/(root)`, bấm `Save`.
    • Chờ 1-2 phút... GitHub sẽ cung cấp một URL.
    • BOOM! Trang web của bạn đã LIVE trên Internet!

TỔNG KẾT KHÓA HỌC

Chúc mừng! Bạn đã đi từ con số 0 đến việc biết cách:
  1. Lưu lịch sử code (`commit`).
  2. Sao lưu code (`push`).
  3. Làm việc an toàn (`branch`).
  4. Làm việc nhóm (`pull request`).
  5. Gỡ rối (`conflict`).
  6. Triển khai một trang web (`clone`, `GitHub Pages`).

Đây là 90% những gì một lập trình viên thực thụ làm với Git hàng ngày!
